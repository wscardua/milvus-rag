## Lacunas conhecidas

- **`python -m app.db.reindex_dynamic_fields` não tem guarda de idempotência** (achado do `/code-review`, PR #12) — enfileira um `ingestion_job=pending` novo para todo documento `indexed`, sem checar se já existe um `pending`/`processing` para aquele documento. Rodar o script duas vezes (ex.: operador tenta de novo porque o worker estava fora do ar) duplica jobs e reprocessamento. Baixo risco (script manual, não corrompe dado — o pipeline é idempotente por documento), mas vale adicionar o guard antes de virar rotina automática — FEAT-INGEST-001, ADR-0015.
- **`vectorstore.sync_document_fields` relê e reescreve todos os chunks do documento (com vetor) pra mudar 1-2 campos de payload** (achado do `/code-review`) — Milvus não tem update parcial de campo, então todo `PATCH` que edita `category_id`/`delivery_phase`/`tags` de um documento com muitos chunks paga o custo de trazer e reenviar todos os vetores. Aceitável na escala da POC; se o acervo crescer, considerar um mecanismo mais barato (ex.: só os campos, se o Milvus passar a suportar) — FEAT-UPLOAD-001, ADR-0015.
- **Conhecimento de "quais campos do `Document` vivem no payload do Milvus" está duplicado sem fonte única** entre `vectorstore.py` (`_PAYLOAD_FIELDS`), `pipeline.py` (linha do upsert na ingestão) e `documents.py` (bloco `milvus_fields` do `PATCH`) — achado do `/code-review`. Hoje mantido por comentário manual; se um ADR futuro adicionar mais um campo filtrável, é fácil esquecer de atualizar o `PATCH` e o índice ficar obsoleto silenciosamente para edições (mesma classe de bug do fix desta PR). Sem teste que force a cobertura dos 3 pontos — considerar um descritor único (nome, declarado/dinâmico, serializador) se mais campos forem adicionados — FEAT-UPLOAD-001, ADR-0007/0015.
- **Rodar `pytest` (apps/api) com o worker real (`python -m app.worker`) ativo em paralelo pode causar corrida** em testes que fazem upload via `client.post("/documents", ...)`: o upload cria um `ingestion_job=pending` de verdade, e o worker (polling 5s) pode reivindicá-lo durante a execução da suíte, reprocessando o documento de teste "de verdade" (classificação por IA via LM Studio, chunks reais no Milvus) por cima do que o teste esperava manipular manualmente. Mitigado nos testes que simulam "documento já indexado" com `_cancel_pending_job` (`test_delivery_metadata.py`), mas o risco genérico continua para qualquer teste de upload — recomendação operacional: **parar o worker antes de rodar a suíte completa** localmente. `CLAUDE.md` não lista o worker como pré-requisito de `pytest`, então isso é específico de rodar os dois processos ao mesmo tempo por conveniência de teste manual — FEAT-UPLOAD-001/FEAT-INGEST-001.
- Calibrar valores de chunk_size/overlap por doc_type com avaliação de retrieval (golden set de perguntas + recall/precisão) — os defaults são estimativas razoáveis, não valores medidos — FEAT-INGEST-001, ADR-0013.
- **Nomenclatura "tokens" na ingestão** (review WORK-006): `Chunk.token_count` e `IngestStats.tokens_min/avg/max` são contagem de **palavras** (`str.split()`), não tokens do modelo de embeddings; o log operacional ("tokens/chunk") herda o termo. Renomear para "palavras" (ou tokenizar de fato) para não confundir na calibração — FEAT-INGEST-001.
- **Qualidade da descrição de imagens (ADR-0012)** — validado ponta a ponta com PDF real de 13 páginas. **Mitigado** com 3 ajustes (prompt estrito de fidelidade + `VISION_MAX_TOKENS` 800 + `VISION_RENDER_DPI` 200): a alucinação de conteúdo inexistente foi eliminada e as tabelas passaram a ser transcritas completas/corretas. **Resíduo:** o modelo default `gemma-3-4b-it-qat` ainda comete erros de OCR em nomes/células densas (ex.: `Radakian→"Carlton"`) — limite do modelo 4B, aceitável p/ POC. Para máxima fidelidade em tabelas críticas, trocar `VISION_MODEL` por modelo maior/tuned p/ documento (só `.env`) — FEAT-INGEST-001.
- Calibrar limiar de similaridade COSINE para "sem contexto suficiente" — FEAT-QUERY-001 (o `query_log` agora fornece scores/rating/latency para essa calibração).
- Calibrar `retrieval_expired_score_factor` (default 0.5) — fator de rebaixamento de documentos vencidos no retrieval; valor estimado, não medido. Considerar também expor a vigência (`valid_until`/vencido) nas citações da UI — FEAT-QUERY-001, ADR-0014.
- ~~Também no Milvus: `delivery_phase`/`valid_until` **não** entram no payload (filtro de fase é no Postgres; vigência é pós-busca). Se surgir necessidade de filtrar por fase no `/query`, exigirá reindexação para popular o payload — FEAT-QUERY-001, ADR-0014.~~ → **resolvido parcialmente (WORK-010, ADR-0015): `delivery_phase` agora filtra `/query`/`/retrieve` via campo dinâmico no Milvus, sem reindexação obrigatória (recomendada/gradual via `python -m app.db.reindex_dynamic_fields`).** `valid_until`/vigência segue como estava (pós-busca, rebaixamento — ADR-0014, não é filtro).
- Definir limite máximo de tamanho de upload — FEAT-UPLOAD-001.
- ~~Decidir se `query_log` entra na POC~~ → **resolvido (ADR-0011, WORK-003): entra, gravando toda consulta com métricas + feedback.**
- **`system_log` sem retenção/rotação** — a tabela cresce indefinidamente; definir política de expurgo/limite (WORK-003, ADR-0011).
- Definir o **prompt de classificação/resumo** que restringe a saída da IA à taxonomia (`reference/taxonomy.md`) — FEAT-INGEST-001.
- Detalhar payloads/erros dos contratos estendidos na implementação (`organization-admin`, `PATCH` de overrides).
- ~~**Filtro `tags` não é filtro de retrieval** — o contrato `query-and-citations` ainda lista `tags` entre os filtros de `/query`, mas o Milvus não indexa `tags`... e `_map_filters` o ignora. Alinhar o contrato (remover `tags` ou indexá-lo no Milvus) — FEAT-QUERY-001/FEAT-MCP-001 (WORK-004).~~ → **resolvido (WORK-010, ADR-0015): `tags` agora é filtro real de `/query`/`/retrieve`/`GET /documents`/tools MCP, semântica OR, via campo dinâmico no Milvus + `overlap` no Postgres (GIN).**
- **`tags` não é editável via `PATCH /documents/{id}`** — só é definida no upload (`DocumentUpdate` não inclui `tags`). Por isso `vectorstore.sync_document_fields` (WORK-010, fix do PATCH) não precisou cobrir `tags`: sem endpoint de edição, não há como ficar obsoleta por essa via. Se um dia `tags` virar editável no detalhe do documento, o `PATCH` precisa ganhar a mesma sincronização já implementada para `category_id`/`delivery_phase` — FEAT-UPLOAD-001.
- **Semântica AND para `tags`** — o filtro de `tags` (`/query`, `/retrieve`, `GET /documents`, tools MCP) só suporta **OR** (documento com qualquer uma das tags pedidas); AND (documento com todas) ficou fora de escopo do WORK-010, decisão registrada no ADR-0015. Reavaliar se surgir demanda — não exige mudança de schema, só trocar `or`/`overlap` por `and`/`contains` nas duas camadas (Milvus `LIKE` + Postgres `@>`).
- **Reindexação do acervo (WORK-010, ADR-0015) é manual, não roda sozinha** — documentos indexados **antes** desta mudança só ganham `delivery_phase`/`tags` no payload do Milvus depois que alguém rodar `python -m app.db.reindex_dynamic_fields` (enfileira `ingestion_job=pending` para todo documento `indexed`) **com o worker ativo** (`python -m app.worker`) para processar a fila. Até lá, filtrar por `delivery_phase`/`tags` nesses documentos antigos simplesmente não os encontra (busca sem esses filtros continua normal — ADR-0015, "recomendado, não bloqueante"). Nenhum agendamento automático foi criado — decisão consciente da POC.
- **UI de Consulta não persiste os filtros selecionados após o POST** (`squad`/`delivery_process`/`category`/`doc_type`/`delivery_phase`/`tags` voltam desmarcados na resposta renderizada) — comportamento pré-existente para os filtros antigos (squad/category/doc_type), não é regressão do WORK-010, mas os 3 filtros novos herdam a mesma limitação. Melhoria futura: reter os valores enviados no `request.POST` ao re-renderizar o form.

## Mapa de integração

- Django (`apps/web/`) → FastAPI (`apps/api/`): contratos `upload-and-metadata` (+file/delete), `query-and-citations` (+feedback), `organization-admin`, `document-links`, `logs-and-health`. Arquivo de documento servido por **proxy** Django.
- FastAPI → PostgreSQL: `squad`, `delivery_process`, `category`, `subcategory`, `document`, `document_link`, `chunk`, `ingestion_job`, `query_log`, `system_log`.
- FastAPI → Milvus: coleção de vetores (`vector[768]` + `chunk_id` + payload `document_id`/`doc_type`/`tags`/`author`/`squad_id`/`delivery_process_id`/`category`/`subcategory`). Container (`ops/`, Podman).
- FastAPI → PostgreSQL: container (`ops/`, Podman).
- FastAPI → LM Studio (API OpenAI-compatível, `base_url` configurável): embeddings `embeddinggemma-300m` (`/v1/embeddings`) e geração (`/v1/chat/completions`).
- Worker (`apps/api/app/worker.py`) → fila `ingestion_job` (Postgres) + LM Studio (embeddings em lote) + Milvus (indexação).
- MCP (`apps/mcp/`) → API FastAPI (`POST /query`, `POST /retrieve`, `GET /documents[...]`) como cliente HTTP (stdio; `API_BASE_URL` default 8001).
